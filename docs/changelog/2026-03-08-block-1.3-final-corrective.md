# 2026-03-08 — Block 1.3 final corrective wave (persisted UI texts + import semantic split)

## Context
This wave closes two residual gaps that still blocked honest completion of Block 1.3:
1. persisted `ui_texts` could override canonical UI text after hydrate;
2. import model still semantically collapsed `Раздел` and `Статья затрат` into one source value.

## Why block 1.3 was still not closed
- Canonical fallbacks in React were already fixed, but persisted DB values (`ui_texts`) could still return legacy text (notably `nav.catalog = Номенклатура`) and visually roll back the UI after `/api/ui-texts` hydration.
- Import pipeline (`types -> parse -> validate -> apply`) still used one field (`purposeCode`) to feed both `defaultPurposeId` and `defaultExpenseArticleId`, violating canonical split between two analytic axes.

## Root cause: persisted ui_texts override
- `seedDefaults()` used `upsert(..., update: {})`, so existing keys were never repaired.
- Existing environments with stale records could keep legacy values forever.
- UI provider correctly hydrated from API, but API was returning stale persisted records.

## Root cause: import semantic collapse
- Import row had one field (`purposeCode`) for both axes.
- Service created/upserted `expenseArticle` and `purpose` from the same code list.
- Item write path and opening lines mapped both ids from the same source value.

## Baseline findings before fix
- `prisma/seed.ts`: no update path for existing `ui_texts` keys.
- `src/lib/import/types.ts`: single `purposeCode` field in directory row.
- `src/lib/import/xlsx/parse.ts`: parsed only one value for analytics.
- `src/lib/import/xlsx/validate.ts`: validated one field and counted it for both summary counters.
- `src/lib/application/import/service.ts`: wrote both ids from one field.

## Fix details: ui_texts synchronization
- Extracted canonical UI text source into `src/lib/ui-texts/defaults.ts`.
- Added explicit repair policy (`shouldRepairUiTextToCanonical`) for known legacy text values (currently includes `nav.catalog: Номенклатура -> Каталог позиций`).
- Implemented idempotent synchronizer `syncCanonicalUiTexts(prisma, mode)` in `src/lib/ui-texts/sync.ts`.
  - `legacy-only` mode: repairs legacy values + scope mismatches only.
  - `force` mode: enforces canonical text/scope for all canonical keys.
- Updated `seedDefaults()` to run canonical sync instead of create-only upserts.
- Added explicit operational script: `npm run ui-texts:sync [legacy-only|force]`.

### Keys synchronized
All canonical keys from `canonicalUiTexts` are included; critical navigation key confirmed:
- `nav.catalog`
- plus other nav and tooltip keys used in hydrated UI.

## Fix details: import semantic split
- Replaced collapsed field:
  - `purposeCode` -> split canonical fields:
    - `sectionCode` (`Раздел` axis)
    - `expenseArticleCode` (`Статья затрат` axis)
- Updated parse layer (`parse.ts`) to read both axes separately.
- Updated validate layer (`validate.ts`):
  - required checks now independent for both axes;
  - summary counters now calculated independently (`purposes` from `sectionCode`, `expenseArticles` from `expenseArticleCode`).
- Updated apply layer (`service.ts`):
  - expense article directory is built from `expenseArticleCode`;
  - purpose directory is built from `sectionCode`;
  - item mappings now use separate source fields:
    - `defaultExpenseArticleId <- expenseArticleCode`
    - `defaultPurposeId <- sectionCode`
  - opening lines now also map separate ids from separate fields.

## Backward compatibility decisions
- Legacy header aliases remain compatibility-only in parser:
  - `Номенклатура` -> `Позиция учёта`
  - `Назначение` -> `Раздел`
  - `Статья расходов` -> `Статья затрат`
- Compatibility does not collapse axes anymore; aliases map into canonical split fields.
