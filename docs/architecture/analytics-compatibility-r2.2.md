# R2.2: compatibility-карта аналитических осей (legacy -> канон)

> Цель: зафиксировать безопасный переход от исторических связей `Item.defaultExpenseArticle/defaultPurpose` к явным каноническим осям `Позиции учёта`.

## 1. Принцип R2.2

- На этапе R2.2 persistence-структура не переименовывается массово.
- Каноническое разведение осей делается в domain-layer через `AccountingPosition.analytics`.
- Legacy-поля считаются источником совместимости, а не каноническим языком предметной модели.

## 2. Соответствие осей

### 2.1 Статья затрат

- **Канон:** `analytics.expenseArticle`.
- **Смысл:** бухгалтерско-складская аналитика позиции.
- **Legacy-источник:** `Item.defaultExpenseArticle`.
- **Дополнительно:** предусмотрено двухуровневое рабочее именование (`workspaceNaming.level1/level2`) без подмены бухгалтерской сущности.

### 2.2 Раздел

- **Канон:** `analytics.section`.
- **Смысл:** рабочий операционный контекст позиции.
- **Legacy-источник:** `Item.defaultPurpose`.
- **Оговорка:** не трактуется как финансовая ось, место хранения или свободный тег.

### 2.3 Управляемые параметры

- **Канон:** `analytics.controlledParameters`.
- **Смысл:** расширяемый control-plane слой поверх базовых осей.
- **Статус R2.2:** структура зафиксирована, полный жизненный цикл управления отложен на R2.3+.

## 3. Контракт доступности

`analytics.availability` определяет применимость осей:

- `required` — ось обязательна для данного режима пространства;
- `optional` — ось допускает отсутствие без разрушения учётного контура;
- `disabled` — ось выключена управляемой конфигурацией.

На R2.2:

- базовые оси выставляются как `required` по умолчанию;
- управляемые параметры — `disabled` по умолчанию;
- жёсткие проверки допустимости состояний будут усилены в R2.3.

## 4. Переходные alias

Для обратной совместимости в канонической структуре остаётся блок `analytics.compatibility`:

- `expenseArticleId`;
- `purposeId`.

Эти поля нужны как временный мост для потребителей, которые ещё не перешли на явные оси `expenseArticle/section`.
