# Карта read-model и проекций vNext

Связанные документы:

- ADR-004: [../adr/ADR-004-writeflow-readmodel.md](../adr/ADR-004-writeflow-readmodel.md)
- Доменная карта: [./domain-model.md](./domain-model.md)
- Карта инвариантов: [./invariants.md](./invariants.md)

## 1. Каноническое определение read-model и проекции

- **Read-model** — модель чтения, оптимизированная под конкретный сценарий просмотра, анализа или управления.
- **Проекция** — производное представление, которое строится из канонических изменений и не может быть самостоятельным источником истины.
- Любая read-model живёт только как следствие канонического write-flow.

## 2. Классы проекций

1. **Рабочие** — оперативные представления для ежедневных действий сотрудников.
2. **Состояния пространства** — текущий операционный срез системы.
3. **Исторические** — последовательность подтверждённых событий и контекстов.
4. **Аналитические** — агрегаты и срезы для управления и оценки.
5. **Управляющие** — представления control plane для конфигурации и контроля.
6. **Сигнальные** — алерты, индикаторы консистентности и operational signals.

## 3. Обязательные базовые проекции vNext

### 3.1 Проекция «Движения»

- **Источник:** канонический слой событий (Движение + строки + контекст).
- **Продуктовая задача:** быстрый операционный цикл создания, отмены и коррекции событий.
- **Тип:** рабочая проекция.
- **Обновление:** событие фиксируется в write-flow, затем проекция обновляется как производная.
- **Недопустимо:**
  - хранить в проекции «черновую истину», не подтверждённую write-flow;
  - редактировать историю движения напрямую в проекции.

### 3.2 Проекция «Склад»

- **Источник:** канонические движения и подтверждённые учётные события.
- **Продуктовая задача:** показывать актуальное состояние пространства по позициям.
- **Тип:** проекция состояния пространства.
- **Обновление:** пересчёт/инкремент только после канонического commit.
- **Недопустимо:** вручную править остатки в обход событийного слоя.

### 3.3 Проекция «История»

- **Источник:** канонический журнал событий и их аудитный контекст.
- **Продуктовая задача:** прозрачная трассировка действий и восстановление причинно-следственной цепочки.
- **Тип:** историческая проекция.
- **Обновление:** append-only семантика с регламентированными компенсирующими действиями.
- **Недопустимо:** «тихое удаление» исторических фактов.

### 3.4 Проекция «Каталог позиций»

- **Источник:** канонические сущности Позиции учёта + классификационные атрибуты.
- **Продуктовая задача:** операционный доступ к позициям, фильтрация и выбор.
- **Тип:** рабочая проекция.
- **Обновление:** после write-flow сущностей и синхронизации с классификаторами.
- **Недопустимо:** дублировать независимую модель позиции вне core domain.

### 3.5 Базовые отчёты

- **Источник:** канонические события + валидные аналитические оси.
- **Продуктовая задача:** управленческая аналитика и интерпретация состояния.
- **Тип:** аналитические проекции.
- **Обновление:** регламентированная пересборка/агрегация на основе подтверждённых данных.
- **Недопустимо:**
  - считать отчёты самостоятельной истиной;
  - включать данные из отключённых неключевых аналитик как обязательные.
- **R2.2 контракт:** аналитические проекции должны учитывать `availability` осей (`required`/`optional`/`disabled`) и корректно деградировать при отключённых неключевых параметрах.

### 3.6 Управляющие представления административной панели

- **Источник:** контур control plane (роли, настройки модулей, feature flags, статусы).
- **Продуктовая задача:** безопасная настройка и контроль поведения системы.
- **Тип:** управляющие проекции.
- **Обновление:** через управляющие write-flow с обязательным аудитом.
- **Недопустимо:** прямой обход канонических ограничений ядра через админ-UI.

## 4. Общие требования к обновлению проекций

1. Проекции обновляются детерминированно по каноническим изменениям.
2. Ошибки проекций должны приводить к диагностике и recovery-процедурам, а не к ручному «ремонту таблиц».
3. Отсутствие неключевого аналитического слоя — допустимое состояние, проекция обязана адаптироваться к нему.
4. Любая рассогласованность между каноном и read-model считается дефектом консистентности.


## 5. Eligibility-контракт R2.3 для расширенных метрик

Для канонической `Позиции учёта` введён базовый policy-признак `projectionEligibility.expandedMetrics` (domain-layer).

- `true`: позиция участвует в расширенных метриках/проекциях;
- `false`: позиция остаётся допустимой в рабочем контуре, но не должна автоматически включаться в расширенные срезы, где отключённый/пустой неключевой слой обязателен по логике отчёта.

Это закрывает требование «допустимое отсутствие неключевой аналитики без разрушения рабочего пространства» и готовит R3 pipeline-проекций.


## 6. R3.2 projection hook (минимальный контракт)

R3.2 добавляет в результат write-flow событийного слоя обязательный payload `projection`:

- `projectionKinds`: `stock | history | reports | signals`;
- `eventType`: тип канонического события;
- `analyticsImpact`: `full | stock_only`;
- `itemIds`, `transactionId` для downstream-проекций.

Это не полный движок проекций, а формальный контракт handoff из write-side в read-side (основа для R3.3/R3.4).


## 7. Кодовая привязка R3.3 (базовый read-side слой)

В R3.3 введён явный слой `src/lib/read-models/*` и обязательные базовые проекции:

- `catalog-projection` — канонический каталог позиций с analytics availability/compatibility/eligibility.
- `stock-projection` — state-проекция текущего склада (без самостоятельной истины).
- `history-projection` — историческая проекция ядра событий с различением `movement/opening/inventory_apply`.
- `consumption-report-projection` — базовый аналитический срез отчётности с фильтрацией по `projectionEligibility.expandedMetrics`.
- `admin-control-projection` — управляющее представление control plane + read-model receipts/recovery contract.

Связка write→read закреплена через `registerProjectionUpdate(...)` на основе payload `projection` из R3.2 use-case результата.

## 10. R3.4 consistency protection / rebuild execution

Для touched area в R3.4 введён исполняемый recovery-контур поверх R3.3 foundation:

- `reset`: очистка projection receipts как подготовка слоя производных состояний к пересборке;
- `re-sync`: повторная синхронизация receipts по каноническому ядру (`catalog`, `stock`, `history`, `reports`, `admin`, `signals`);
- `consistency checker`: верификация receipts против канона + сигнализация reduced eligibility как допустимого informational состояния.

Таким образом read-side получил не только контракты handoff из write-side, но и управляемый путь repair/rebuild.
