# Архитектурная карта хаба «Движения»

Связанные документы:

- ADR-006: [../adr/ADR-006-movements-hub.md](../adr/ADR-006-movements-hub.md)
- Доменная карта: [./domain-model.md](./domain-model.md)
- Карта write-flow: [./write-flow-map.md](./write-flow-map.md)

## 1. Роль хаба «Движения» в архитектуре

Хаб «Движения» — главный рабочий контекст сотрудников. Он предназначен для быстрого выполнения ежедневных учётных действий в едином цикле: подготовка -> фиксация события -> подтверждение результата -> следующий шаг.

Хаб не является отдельной «логикой учёта». Он является операционным представлением канонического write-flow учётных событий.

## 2. Рабочее поле как центральная зона

- Центральная зона хаба — рабочее поле движения.
- В рабочем поле пользователь подбирает позиции, указывает количество, контекст и аналитики.
- Результат действия должен быть однозначен и сразу подтверждать, что событие зафиксировано в каноне.

## 3. Рабочая строка позиции как базовая единица

- Базовая единица взаимодействия — рабочая строка позиции (или компактная карточка в мобильной форме).
- Строка содержит минимум, необходимый для канонического события:
  - позиция;
  - количество;
  - единица;
  - контекст (раздел и доступные аналитики).
- Строка не может переходить в фиксацию в «полусобранном» виде.

## 4. Серийный/массовый режим

- В массовом режиме заполненная строка означает участие в действии.
- Неполные строки не считаются участниками commit и не должны создавать «тихие» артефакты.
- Подтверждение серии должно формировать канонический batch событийного изменения с полной валидацией.

## 5. Локальное раскрытие аналитик и параметров

- Дополнительные аналитики и параметры раскрываются локально по строке/контексту.
- Обязательные аналитики проверяются до commit.
- Неключевые аналитики могут быть отключены конфигурацией без блокировки базового сценария.

## 6. Встроенный выбор единицы

- Выбор единицы измерения встроен в рабочую строку как доменный атрибут, а не декоративная опция UI.
- Конвертации и допустимые значения подчиняются классификационному слою и инвариантам write-flow.

## 7. Post-action цикл

После успешного действия хаб остаётся в рабочем режиме:

1. показывает ясный результат операции;
2. обновляет рабочее состояние (проекция «Движения» и связанные представления);
3. готовит пользователя к следующему действию без перехода в «разрывной» сценарий.

## 8. Состояния и режимы хаба

- **Готов к вводу** — базовый рабочий режим.
- **Подготовка пакета** — формирование локального набора строк до commit.
- **Валидация/фиксация** — транзитное состояние write-flow.
- **Успешное завершение** — подтверждение события и переход в post-action.
- **Управляемая ошибка** — явное сообщение + безопасный путь исправления.

Недопустимо состояние, в котором пользователь должен вручную «чинить» консистентность системы.

## 9. Desktop и mobile как две формы одного канона

- Desktop и mobile — разные поведенческие формы одного канонического сценария.
- Различаться могут плотность интерфейса и навигационные жесты.
- Не могут различаться:
  - бизнес-семантика события;
  - набор обязательных проверок;
  - порядок канонического write-flow.

## 10. Границы ответственности хаба

1. Хаб отвечает за эффективный операционный ввод и прозрачную обратную связь.
2. Хаб не хранит отдельную предметную истину.
3. Все фиксации проходят через канонический write-flow и отражаются в read-model как производные изменения.

## 11. Реализация R5.1 (первая живая версия экрана)

На этапе R5.1 канон получил рабочую UI-реализацию без wholesale-переписывания:

- канонический route хаба: `/movements`;
- legacy `/operation` сохранён как compatibility redirect;
- верхний слой экрана = рабочий контекст (тип движения, дата, режим прихода, раздел);
- центральный слой = рабочее поле позиций, которое загружается сразу в контексте раздела;
- поиск оставлен как вспомогательный ускоритель и сужает уже существующее поле;
- action слой представлен отдельной панелью по выбранной позиции и готов к развитию в рабочую строку R5.2.

Data-path R5.1:

1. UI вызывает `fetchMovementWorkspace(sectionId, q)`;
2. client читает `/api/items` с `purposeId`;
3. route отдаёт проекцию `getPositionCatalogProjection(...)` (read-model слой);
4. хаб формирует рабочее поле и сценарий подготовки строки движения.

Это закрепляет принцип «рабочее пространство прежде формы» и подготавливает следующие шаги (R5.2+), не ломая канонический write/read контур.


## 12. Реализация R5.2 (рабочая строка + action-ready слой)

Этап R5.2 переводит хаб от compatibility-панели к прямому действию в строке:

- каждая позиция в `Рабочем поле` — самостоятельная action-card (идентичность + количество + единица + action control);
- ввод количества и выбор единицы выполняются в контексте строки без перехода в heavy form;
- commit позиции уходит в уже существующий write-side (`POST /api/transactions`) с канонической семантикой `IN / OUT / ADJUST`;
- строка имеет локальные error/success состояния, которые не ломают соседние строки;
- локальное раскрытие `Доп. параметры` оставляет первый слой чистым и подготавливает второй слой аналитик.

Compatibility-ограничение R5.2: распределённый приход (`IN + DISTRIBUTE_PURPOSES`) пока использует batch-линии как переходный контур, чтобы не блокировать будущий R5.3 серийного режима.

## 13. Реализация R5.3 (серийный режим без checkbox-selection)

Этап R5.3 закрепляет канонический серийный режим внутри того же хаба:

- состояние хаба поддерживает **несколько активных строк одновременно** (`multi-row drafts`);
- участие строки определяется **вводом количества**, а не отдельным выбором;
- submit-path работает по правилу `filled rows only` и уходит в существующий write-side контракт.

Ключевые решения:

1. **Multi-row state**
   - `rowDrafts` хранит изолированные draft-состояния по `itemId`;
   - поздняя гидратация единиц (`fetchItemUnits`) не сбрасывает введённый `qtyInput`;
   - выбор единицы и дополнительные поля остаются локальными для каждой строки.

2. **Правило участия (`filled = participate`)**
   - строка считается участником только если `qtyInput` непустой;
   - пустые строки не попадают в submit payload;
   - отдельные чекбоксы/корзина выбора не используются.

3. **Serial/batch-ready apply path**
   - единая команда `Провести заполненные строки` собирает все participating rows;
   - перед submit выполняется локальная валидация каждой участвующей строки;
   - при ошибках submit блокируется, а блокирующие строки получают локальные сообщения;
   - при успехе формируется единый payload в `POST /api/transactions`.

4. **Совместимость с одиночным сценарием**
   - одна заполненная строка проходит тем же путём без переключения режима;
   - пользовательский ритм «быстро провести одну позицию» сохранён.

5. **Foundation для R5.4**
   - после submit явно известно, какие строки участвовали;
   - результат и ошибки разведены по уровням (row-local / submit-global);
   - архитектура готова к наращиванию post-action цикла и локальной коррекции без перепроектирования state-модели.

## 14. Реализация R5.4 (post-action цикл внутри хаба)

R5.4 закрывает обязательный цикл `действие → понятный результат → короткий rollback/correction path → продолжение работы` без выпадения в отдельный продуктовый режим.

Ключевые элементы:

1. **Локальный result layer (не отдельный экран)**
   - успех submit материализуется в `PostActionState`;
   - слой рендерится внутри того же хаба (`ResultView`), рядом с рабочим полем;
   - отображает batch + список проведённых строк + summary по режиму (`single`/`multi`).

2. **Локальный rollback недавнего действия (touched safe scope)**
   - быстрый путь из result layer: `Локально отменить последнее действие`;
   - UI вызывает `POST /api/transactions/[id]/rollback`, который делегирует в `createRecoveryService().rollbackMovement(...)`;
   - применяются ограничения канона R3.4: movement rollback (`IN/OUT/ADJUST`), история сохраняется, destructive universal recovery UI не открывается.

3. **Локальная коррекция недавнего действия**
   - path `Исправить в рабочем поле` переносит данные строки результата обратно в текущий row draft (`rehydrate`);
   - пользователь остаётся в том же разделе и сразу может повторно провести корректировку;
   - fallback-path `Исправить в модальном редакторе` сохранён для scoped compatibility.

4. **Согласование с multi-row foundation R5.3**
   - result layer строится по `participatingItemIds`, а не по single-row assumptions;
   - при multi-row submit пользователь видит количество реально проведённых строк и line-level детали;
   - рабочее поле не сбрасывается wholesale, что сохраняет серийный ритм.

5. **Разделение state-уровней**
   - row-level validation: `rowDrafts[itemId].error`;
   - workspace submit failure: `workspaceError`;
   - success/result: `postAction`;
   - rollback/correction feedback: toast + локальная ошибка.

Это создаёт устойчивый foundation для следующего слоя: richer local details (второй слой аналитик, компактная недавняя лента, deeper batch correction matrix) без смены архитектуры хаба.

## 15. Реализация R5.5 (локальный второй слой строки/карточки)

R5.5 закрепляет второй слой как **локальное раскрытие из рабочей строки**, а не как отдельный режим экрана.

Ключевые решения:

1. **Локальное раскрытие без выпадения из workspace**
   - в каждой строке добавлен отдельный триггер `2-й слой`;
   - раскрытие происходит inline в той же карточке позиции;
   - раскрытие одной строки не перестраивает весь экран и не влияет на соседние строки.

2. **Чистый первый слой (action-ready)**
   - первый слой оставляет только базовые controls: позиция, количество, единица, `Провести`;
   - второй слой не переносится в primary controls и не конкурирует с main action.

3. **Состав второго слоя**
   - аналитики второго слоя (section/expenseArticle) отображаются локально в контексте позиции;
   - controlledParameters показаны как extension point c `availability` + `valuesCount` семантикой;
   - вторичные действия (toggle hints / toggle controlledParameters / reset row draft) вынесены во второй слой;
   - контекстные eligibility/availability пояснения отображаются коротко и спокойно (не как аварийные ошибки).

4. **Согласование с serial mode и post-action**
   - second-layer UI-state хранится в row draft (`secondLayerExpanded`, `showEligibilityHint`, `showControlledParameters`), поэтому serial participation logic остаётся инвариантной;
   - правило `filled rows only` не затронуто;
   - post-action correction (`rehydrate`) совместим со вторым слоем: строка остаётся редактируемой и может быть уточнена локально.

5. **Foundation для следующего refinement-шага**
   - реализация не привязана к глобальной панели/модалке и допускает mobile/desktop полировку;
   - второй слой реализован как расширяемый row-local контейнер без смены write/read/recovery архитектуры.
