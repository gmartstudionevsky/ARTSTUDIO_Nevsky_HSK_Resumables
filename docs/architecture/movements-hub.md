# Архитектурная карта хаба «Движения»

Связанные документы:

- ADR-006: [../adr/ADR-006-movements-hub.md](../adr/ADR-006-movements-hub.md)
- Доменная карта: [./domain-model.md](./domain-model.md)
- Карта write-flow: [./write-flow-map.md](./write-flow-map.md)

## 1. Роль хаба «Движения» в архитектуре

Хаб «Движения» — главный рабочий контекст сотрудников. Он предназначен для быстрого выполнения ежедневных учётных действий в едином цикле: подготовка -> фиксация события -> подтверждение результата -> следующий шаг.

Хаб не является отдельной «логикой учёта». Он является операционным представлением канонического write-flow учётных событий.

## 2. Рабочее поле как центральная зона

- Центральная зона хаба — рабочее поле движения.
- В рабочем поле пользователь подбирает позиции, указывает количество, контекст и аналитики.
- Результат действия должен быть однозначен и сразу подтверждать, что событие зафиксировано в каноне.

## 3. Рабочая строка позиции как базовая единица

- Базовая единица взаимодействия — рабочая строка позиции (или компактная карточка в мобильной форме).
- Строка содержит минимум, необходимый для канонического события:
  - позиция;
  - количество;
  - единица;
  - контекст (раздел и доступные аналитики).
- Строка не может переходить в фиксацию в «полусобранном» виде.

## 4. Серийный/массовый режим

- В массовом режиме заполненная строка означает участие в действии.
- Неполные строки не считаются участниками commit и не должны создавать «тихие» артефакты.
- Подтверждение серии должно формировать канонический batch событийного изменения с полной валидацией.

## 5. Локальное раскрытие аналитик и параметров

- Дополнительные аналитики и параметры раскрываются локально по строке/контексту.
- Обязательные аналитики проверяются до commit.
- Неключевые аналитики могут быть отключены конфигурацией без блокировки базового сценария.

## 6. Встроенный выбор единицы

- Выбор единицы измерения встроен в рабочую строку как доменный атрибут, а не декоративная опция UI.
- Конвертации и допустимые значения подчиняются классификационному слою и инвариантам write-flow.

## 7. Post-action цикл

После успешного действия хаб остаётся в рабочем режиме:

1. показывает ясный результат операции;
2. обновляет рабочее состояние (проекция «Движения» и связанные представления);
3. готовит пользователя к следующему действию без перехода в «разрывной» сценарий.

## 8. Состояния и режимы хаба

- **Готов к вводу** — базовый рабочий режим.
- **Подготовка пакета** — формирование локального набора строк до commit.
- **Валидация/фиксация** — транзитное состояние write-flow.
- **Успешное завершение** — подтверждение события и переход в post-action.
- **Управляемая ошибка** — явное сообщение + безопасный путь исправления.

Недопустимо состояние, в котором пользователь должен вручную «чинить» консистентность системы.

## 9. Desktop и mobile как две формы одного канона

- Desktop и mobile — разные поведенческие формы одного канонического сценария.
- Различаться могут плотность интерфейса и навигационные жесты.
- Не могут различаться:
  - бизнес-семантика события;
  - набор обязательных проверок;
  - порядок канонического write-flow.

## 10. Границы ответственности хаба

1. Хаб отвечает за эффективный операционный ввод и прозрачную обратную связь.
2. Хаб не хранит отдельную предметную истину.
3. Все фиксации проходят через канонический write-flow и отражаются в read-model как производные изменения.

## 11. Реализация R5.1 (первая живая версия экрана)

На этапе R5.1 канон получил рабочую UI-реализацию без wholesale-переписывания:

- канонический route хаба: `/movements`;
- legacy `/operation` сохранён как compatibility redirect;
- верхний слой экрана = рабочий контекст (тип движения, дата, режим прихода, раздел);
- центральный слой = рабочее поле позиций, которое загружается сразу в контексте раздела;
- поиск оставлен как вспомогательный ускоритель и сужает уже существующее поле;
- action слой представлен отдельной панелью по выбранной позиции и готов к развитию в рабочую строку R5.2.

Data-path R5.1:

1. UI вызывает `fetchMovementWorkspace(sectionId, q)`;
2. client читает `/api/items` с `purposeId`;
3. route отдаёт проекцию `getPositionCatalogProjection(...)` (read-model слой);
4. хаб формирует рабочее поле и сценарий подготовки строки движения.

Это закрепляет принцип «рабочее пространство прежде формы» и подготавливает следующие шаги (R5.2+), не ломая канонический write/read контур.
