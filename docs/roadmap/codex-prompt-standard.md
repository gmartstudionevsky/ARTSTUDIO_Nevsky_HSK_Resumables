# Codex Prompt Standard (канонический стандарт постановки задач)

> Версия: 2026-03-09.
>
> Это обязательный стандарт подготовки всех последующих Codex-промтов по проекту ARTSTUDIO Consumables.
>
> Ни один существенный шаг реализации, рефакторинга, документационной интеграции или выравнивания repo не должен запускаться вне этого стандарта.
>
> Стандарт введён для защиты проекта от формальной, поверхностной и нетрассируемой реализации.
>
> Каждый будущий промт обязан быть связан с:
> - `docs/roadmap/master-plan.md`;
> - `docs/roadmap/canon-traceability-map.md`;
> - релевантными каноническими документами `docs/product/canon/*`;
> - `docs/roadmap/progress-log.md`.

## 1. Статус стандарта

- Статус: `active, mandatory, execution-gating`.
- Область применения: все значимые Codex-задачи (разработка, рефакторинг, docs-интеграция, выравнивание repo).
- Режим обязательности: отсутствие обязательных блоков промта означает, что задача сформулирована невалидно и не должна запускаться.

## 2. Назначение стандарта

Codex-промт в этом проекте — не запрос «сделай X», а управляемый инструмент реализации канона в репозитории.

Корректный промт обязан предотвращать:

- формальный patching без глубины;
- локальную реализацию без связи с моделью;
- UI-реализацию без model-layer изменений там, где они обязательны;
- docs-only реализацию там, где требуется изменение поведения системы;
- naming cleanup без structural cleanup;
- тесты-ширмы вместо проверки фактического соответствия канону.

## 3. Обязательные принципы Codex-промта

1. **Traceability first**
   - Каждый промт обязан быть привязан к шагу/подшагу master-plan и к конкретным канонам.
2. **Scope clarity**
   - Scope задачи обязан быть ясным, ограниченным и операционно проверяемым.
3. **Deep implementation over visible patch**
   - Промт обязан требовать глубинную реализацию, а не только видимый поверхностный эффект.
4. **Repo-aware execution**
   - Промт обязан учитывать текущую структуру repo и существующие слои, а не абстрактную «идеальную систему».
5. **Acceptance must be testable**
   - Acceptance criteria обязаны быть проверяемыми по артефактам repo и поведению системы.
6. **No checkbox implementation**
   - Промт не должен допускать формальное закрытие пункта без содержательного закрытия канона.
7. **Explicit non-goals**
   - Промт обязан явно фиксировать, что в рамках задачи не делается.

## 4. Структура каждого будущего промта

Каждый значимый Codex-промт обязан содержать блоки строго в этом порядке:

1. Название задачи
2. Тип работы
3. Цель
4. Место в master-plan
5. Каноническое основание
6. Репозиторные точки воздействия
7. Текущее проблемное состояние / gap
8. Почему без этой задачи repo ещё не соответствует канону
9. Что именно нужно сделать
10. Ограничения / что не делать
11. Acceptance criteria
12. Evidence of completion
13. Как именно будет видно, что решение не формальное
14. Признаки поверхностной реализации
15. Ожидаемое обновление progress-log / docs после выполнения

## 5. Обязательные блоки промта

### 5.1 Название задачи
- **Зачем нужно:** фиксирует операционный фокус задачи и исключает размытость формулировки.
- **Что должно быть:** краткое, предметное название без общих слов «улучшить/оптимизировать всё».

### 5.2 Тип работы
- **Зачем нужно:** сразу задаёт рамку (кодовая, документационная, архитектурная, mixed).
- **Что должно быть:** явный тип и допустимый слой изменений.

### 5.3 Цель
- **Зачем нужно:** определяет продуктовый результат, а не перечень действий.
- **Что должно быть:** одно чёткое целевое состояние, проверяемое в конце задачи.

### 5.4 Место в master-plan
- **Зачем нужно:** гарантирует движение по утверждённому маршруту, а не ad-hoc внедрение.
- **Что должно быть обязательно:**
  - указание шага;
  - указание подшага (если существует);
  - указание зависимости от предыдущих шагов (если критично).

### 5.5 Каноническое основание
- **Зачем нужно:** предотвращает постановку задач «вне канона».
- **Что должно быть обязательно:**
  - перечисление конкретных канонов;
  - ссылки на конкретные canonical docs;
  - явное указание, какой canon-gap закрывается.

### 5.6 Репозиторные точки воздействия
- **Зачем нужно:** делает промт repo-aware и технически реалистичным.
- **Что должно быть обязательно:** явный список затрагиваемых слоёв, где применимо:
  - docs;
  - prisma/schema;
  - seed;
  - domain/application;
  - read-models;
  - routes;
  - UI/components;
  - tests;
  - admin/control plane;
  - configs/scripts.

### 5.7 Текущее проблемное состояние / gap
- **Зачем нужно:** связывает задачу с реальным разрывом, а не с абстрактным «улучшением».
- **Что должно быть обязательно:**
  - конкретное описание текущего разрыва;
  - связь с `canon-traceability-map`;
  - пояснение, почему без этого разрыва repo остаётся canon-incomplete.

### 5.8 Почему без этой задачи repo ещё не соответствует канону
- **Зачем нужно:** закрепляет обязательность задачи как закрытия canon-gap.
- **Что должно быть обязательно:**
  - прямое обоснование несоответствия канону без выполнения задачи;
  - указание слоя, где несоответствие проявляется (модель/flow/UI/docs/tests).

### 5.9 Что именно нужно сделать
- **Зачем нужно:** переводит цель в операционно исполнимый набор результатов.
- **Что должно быть обязательно:**
  - конкретный ожидаемый результат;
  - поддействия (если нужны);
  - запрет на расплывчатые формулировки без операционного смысла.

### 5.10 Ограничения / что не делать
- **Зачем нужно:** защищает scope от расползания и имитации широкой реализации.
- **Что должно быть обязательно:**
  - явные non-goals;
  - запрет на затрагивание несвязанных слоёв;
  - запрет на имитацию «большого закрытия» вне заданного scope.

### 5.11 Acceptance criteria
- **Зачем нужно:** задаёт конечные условия приёмки задачи.
- **Что должно быть обязательно:**
  - только проверяемые критерии;
  - критерии по слоям (docs/model/flow/UI/tests при релевантности);
  - запрет на формулировки «примерно реализовано».

### 5.12 Evidence of completion
- **Зачем нужно:** фиксирует, как выполнение проверяется без доверия «на слово».
- **Что должно быть обязательно:** конкретные evidence-следы, например:
  - файлы и изменения в них;
  - обновлённые docs;
  - выровненные schema/adapters/naming;
  - работающий route/flow;
  - тесты нужного уровня;
  - отсутствие legacy-расхождения;
  - подтверждаемое user-facing изменение.

### 5.13 Как именно будет видно, что решение не формальное
- **Зачем нужно:** заранее блокирует shallow-реализацию.
- **Что должно быть обязательно:**
  - признаки глубины по слоям;
  - критерий, что канон закрыт содержательно, а не косметически.

### 5.14 Признаки поверхностной реализации
- **Зачем нужно:** заранее задаёт fail-condition для результата.
- **Что должно быть обязательно:** список конкретных «плохих исходов» именно для данной задачи.

### 5.15 Ожидаемое обновление progress-log / docs после выполнения
- **Зачем нужно:** замыкает operational loop фиксации выполнения.
- **Что должно быть обязательно:**
  - как обновится `progress-log`;
  - какие docs должны быть синхронизированы после выполнения.

## 6. Требования к acceptance criteria

Acceptance criteria обязаны:

1. Быть конечными и проверяемыми.
2. Различать эффекты по уровням: docs-level, model-level, flow-level, UI-level, test-level (где релевантно).
3. Исключать двусмысленность «примерно/частично похоже».
4. Отвечать на вопрос: как понять, что канон реализован глубоко, а не только показан на поверхности.
5. Быть привязаны к шагу master-plan и к конкретному canon-gap.

## 7. Требования к evidence of completion

1. Evidence не дублирует acceptance; evidence фиксирует конкретные следы глубокой реализации.
2. Evidence должен позволять проверку без доверия «на слово».
3. Evidence должен быть repo-based (файлы, контракты, тесты, маршруты, поведение).
4. Evidence должен быть привязан к соответствующему шагу master-plan и канону.
5. Evidence должен быть пригоден для последующего аудита и фиксации в progress-log.

## 8. Что считается поверхностной реализацией

К общим антипаттернам относятся:

- placeholder вместо реального слоя;
- docs without code change там, где требуется код;
- UI without model cleanup;
- rename without semantic migration;
- partial route without coherent flow;
- тест, не проверяющий реальный канон;
- реализация, оставляющая legacy-path фактически основным;
- визуальное наличие функции без product-level integration;
- закрытие acceptance через локальный hack.

## 9. Что считается глубокой реализацией

Глубокая реализация подтверждается, если одновременно выполнено:

1. Изменение укладывается в канон и закрывает конкретный canon-gap.
2. Модель, flows, naming, docs и tests согласованы между собой.
3. Нет скрытого системного противоречия (включая критичный legacy leakage).
4. Решение работает в структуре repo, а не «рядом» с ней.
5. Продукт становится canon-complete глубже, а не просто visually richer.
6. Implementation оставляет проверяемый след для progress-log и будущего аудита.

## 10. Как использовать стандарт вместе с master-plan, traceability map и progress-log

- `master-plan` задаёт маршрут и порядок шагов.
- `canon-traceability-map` связывает каноны с gaps и целевыми шагами.
- `codex-prompt-standard` задаёт обязательное качество постановки каждой задачи.
- `progress-log` фиксирует фактическое продвижение и глубину закрытия.
- Эти четыре документа работают как единый operational loop:
  `канон → traceability map → master-plan → codex prompt → implementation → progress-log`.
