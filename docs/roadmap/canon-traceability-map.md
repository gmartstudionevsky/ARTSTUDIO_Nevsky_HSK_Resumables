# Canon traceability map

> Цель документа: единая рабочая карта трассируемости `канон → продуктовый смысл → артефакты repo → статус реализации → gap → шаг master-plan`.
>
> Базовые источники: `docs/product/canon/*`, `docs/product/spec-vnext.md`, `docs/roadmap/master-plan.md`, `docs/audit/2026-03-08-repo-audit.md`, текущее состояние кода и структуры repo.

## Правила чтения этой карты

- Оценка «реализовано» относится к **содержательной глубине**, а не к факту существования файла/экрана.
- «Частично» означает: есть foundation, но отсутствует сквозная консистентность (термины, UX-ритм, контроль, ошибки, доказуемость).
- «Не реализовано» означает: в repo нет продуктово завершённого слоя, либо он представлен только локальными заготовками.
- Ссылки на шаги плана даны по `docs/roadmap/master-plan.md` (блок + подэтап).

## 1) Базовая десятка канонов

### 1.1 Атом доменного факта

- **Название канона:** Атом доменного факта.
- **Канонический смысл (2–6 строк):** Минимальная неделимая единица истины — зафиксированное движение по позиции учёта с типом, количеством, временем, автором и аналитическими осями. Все производные состояния строятся из этих фактов, а не из отдельных «локальных» таблиц состояния.
- **Какие части repo уже относятся к канону:** `prisma/schema.prisma` (`Transaction`, `TransactionLine`, `RecordStatus`, `TxType`), `src/lib/application/accounting-event/*`, `src/lib/application/accounting-position/*`, API `src/app/api/transactions/*`, тесты `tests/application/accounting-event.write-flow.test.ts`, `tests/domain/accounting-position.invariants.test.ts`.
- **Что уже реализовано содержательно:** Канонические типы событий (`IN/OUT/ADJUST/OPENING/INVENTORY_APPLY`) и неизменяемая журнализация через статусную отмену (`ACTIVE/CANCELLED`) присутствуют в persistence и application-слое.
- **Что реализовано частично / поверхностно / фрагментарно:** Единый продуктовый язык факта ещё не полностью изолирован от legacy-терминов во внутренних именах (`purpose*`, `item*`), что ухудшает читаемость трассировки.
- **Что не реализовано:** Полный product-grade error/diagnostics слой факта для всех критических сценариев (нормализация отказов и сигналов).
- **Какие legacy / compatibility / structural debt мешают:** Legacy-naming в persistence/API (`Item`, `Purpose`) и совместимость старых маршрутов.
- **Шаг утверждённого плана для доведения:** 1.4 (локализация compatibility-layer), 3.6 (нормализация error layer), 9.1 (observability).
- **Какие подшаги логично ожидаются позже:** Договориться о каноническом envelope ошибок write-side; ввести единый trace-id/корреляцию в user/admin потоках.
- **Будущие доказательные артефакты глубокой реализации:** Контракты ошибок write API, интеграционные тесты на отказоустойчивость и корректную диагностику, runbook по расследованию write-side инцидентов.

### 1.2 Доменные оси

- **Название канона:** Доменные оси (Статья затрат + Раздел).
- **Канонический смысл:** Эти две оси задают координатную систему движения; они не являются декоративными справочниками и должны быть сквозными в write/read/UI/reporting.
- **Какие части repo уже относятся к канону:** `prisma/schema.prisma` (`ExpenseArticle`, `Purpose`, ссылки в `Item`/`TransactionLine`), `src/lib/domain/accounting-position/*`, `src/lib/application/accounting-position/*`, import use-case `src/lib/application/import/service.ts`, API lookup/admin dictionaries.
- **Что уже реализовано содержательно:** Разведение сущностей в схеме и в потоках импорта/операций есть; данные сохраняются и читаются раздельно.
- **Что реализовано частично / поверхностно / фрагментарно:** В части UX и copy остаются следы исторического конфликта «Раздел/Назначение»; отдельные внутренние имена продолжают транслировать legacy.
- **Что не реализовано:** Полностью очищенная каноническая терминология во всех слоях без compatibility leakage.
- **Какие legacy / compatibility / structural debt мешают:** Внутренний `purpose*` слой как технический долг миграции именований.
- **Шаг утверждённого плана для доведения:** 1.3–1.4, 2.4, 4.4.
- **Какие подшаги логично ожидаются позже:** Завершить канон-first naming в внутренних контрактах; выровнять фильтры/заголовки/подсказки во всех экранах.
- **Будущие доказательные артефакты глубокой реализации:** Матрица терминов code+UI+docs, регрессионные тесты текстов и API-контрактов без legacy fallback.

### 1.3 Единицы измерения

- **Название канона:** Единицы измерения и конвертация.
- **Канонический смысл:** Позиция ведётся в базовой единице с допустимыми входными/отчётными единицами; операции должны гарантировать корректность конвертации и инварианты выбора дефолтных единиц.
- **Какие части repo уже относятся к канону:** `prisma/schema.prisma` (`Unit`, `ItemUnit`, поля qty*), `src/lib/stock/calc.ts`, `src/lib/items/validators.ts`, `src/lib/inventory/validators.ts`, импорт `src/lib/import/xlsx/validate.ts`, тесты domain/application/e2e вокруг операций и учёта.
- **Что уже реализовано содержательно:** Модель multi-unit и вычисление `qtyBase` присутствуют; дефолтные единицы и allowed-набор валидируются.
- **Что реализовано частично / поверхностно / фрагментарно:** Доказательная часть (отчётные UX-сценарии граничных случаев и полная диагностика ошибок единиц) не оформлена как единый продуктовый контракт.
- **Что не реализовано:** Отдельный операционный контур мониторинга ошибок конвертации и контроль качества данных по единицам.
- **Какие legacy / compatibility / structural debt мешают:** Исторические импортные алиасы и неоднородный слой сообщений ошибок.
- **Шаг утверждённого плана для доведения:** 3.5–3.6, 5.1, 9.1.
- **Какие подшаги логично ожидаются позже:** Ввести единый словарь ошибок unit-conversion; расширить e2e/contract coverage на edge-cases.
- **Будущие доказательные артефакты глубокой реализации:** Отдельные контрактные тесты конвертации + контрольные отчёты качества по единицам.

### 1.4 Временная модель

- **Название канона:** Временная модель.
- **Канонический смысл:** Факт имеет `occurredAt`, `createdAt`, статусные изменения и ограничения по периодам; temporal-поведение должно быть предсказуемым в write/read/recovery.
- **Какие части repo уже относятся к канону:** `prisma/schema.prisma` (временные поля в транзакциях, сессиях, блокировках периода), `src/lib/period-locks/service.ts`, API period-locks, история/склад read-модели.
- **Что уже реализовано содержательно:** Временные поля и базовый контроль периода реализованы; события сортируются и читаются как журнал.
- **Что реализовано частично / поверхностно / фрагментарно:** Сквозная temporal-диагностика, SLA-проверки задержек и единая политика корректировок по времени не собраны в один operational слой.
- **Что не реализовано:** Полный набор эксплуатационных temporal SLO/alerts.
- **Какие legacy / compatibility / structural debt мешают:** Фрагментированность сигналов между приложением и админ-контуром.
- **Шаг утверждённого плана для доведения:** 6.3, 9.1, 9.2.
- **Какие подшаги логично ожидаются позже:** Ввести метрики latency/lag по проекциям и batch-сценариям; закрепить runbook temporal-инцидентов.
- **Будущие доказательные артефакты глубокой реализации:** Набор health/extended метрик, регулярный отчёт по задержкам, тесты на period-lock policy.

### 1.5 Жизненный цикл движения

- **Название канона:** Жизненный цикл движения (создание, отмена, коррекция, последствия).
- **Канонический смысл:** Движение не удаляется физически; управляется статусами, причинами отмены, исправлениями строк и восстановлением согласованности проекций.
- **Какие части repo уже относятся к канону:** `src/app/api/transactions/*`, `src/app/api/transaction-lines/*`, `src/lib/application/accounting-event/*`, `src/lib/application/recovery/*`, tests `tests/components/operation.*`, `tests/application/recovery.use-cases.test.ts`.
- **Что уже реализовано содержательно:** Отмена batch/line и коррекция строки есть, recovery foundation присутствует, проекции получают receipt-обновления.
- **Что реализовано частично / поверхностно / фрагментарно:** UX-state story post-action/undo/correction ещё не доведена до полного product-grade уровня.
- **Что не реализовано:** Универсальный recovery engine для всех специальных сценариев и полный единый user-facing слой ошибок/подсказок.
- **Какие legacy / compatibility / structural debt мешают:** Частичная локальность recovery-сценариев и неполная унификация сообщений отказа.
- **Шаг утверждённого плана для доведения:** 3.5–3.6, 4.6.
- **Какие подшаги логично ожидаются позже:** Единые сценарии correction/undo в списках и деталях, расширенные recovery-actions в control plane.
- **Будущие доказательные артефакты глубокой реализации:** Сквозные e2e-cases жизненного цикла + админ-действия восстановления с аудитом.

### 1.6 Специальные режимы действия

- **Название канона:** Специальные режимы (`OPENING`, `INVENTORY_APPLY`, import modes).
- **Канонический смысл:** Спецрежимы не должны маскироваться под обычные IN/OUT; они имеют самостоятельную семантику и отдельные правила отображения/контроля.
- **Какие части repo уже относятся к канону:** `TxType` в `prisma/schema.prisma`, инвентаризация `src/app/api/inventory/*`, импорт `src/lib/application/import/service.ts`, история и read-model проекции.
- **Что уже реализовано содержательно:** Типы присутствуют в модели; импорт умеет управлять opening mode; инвентаризация имеет отдельные статусы/сущности.
- **Что реализовано частично / поверхностно / фрагментарно:** Сквозная user-facing дифференциация спецрежимов в каждом представлении и каналах уведомлений неполная.
- **Что не реализовано:** Полный контракт отображения/аналитики спецрежимов во всех отчётах и интеграционных сообщениях.
- **Какие legacy / compatibility / structural debt мешают:** Исторические сценарии, где opening трактовался как обычный приход; фрагментарные шаблоны интеграций.
- **Шаг утверждённого плана для доведения:** 3.5, 5.2, 5.3, 8.1.
- **Какие подшаги логично ожидаются позже:** Выделить отдельные визуальные и отчётные маркеры для OPENING/INVENTORY_APPLY; расширить Telegram/event templates.
- **Будущие доказательные артефакты глубокой реализации:** Тесты истории/отчётов/уведомлений по спецрежимам и канонический справочник event semantics.

### 1.7 Строка позиции учёта

- **Название канона:** Строка позиции учёта как операционная единица ввода.
- **Канонический смысл:** Рабочая единица хаба — строка действия по позиции (количество, единица, аналитики, комментарий, статус post-action), а не «форма ради формы».
- **Какие части repo уже относятся к канону:** UI `src/components/operation/*`, route `src/app/(app)/movements/page.tsx`, API operation/transactions, tests `tests/components/operation.*`, `tests/e2e/core-flow.spec.ts`.
- **Что уже реализовано содержательно:** Row-level actions, batch lines и post-action state реализованы как foundation; есть модальные действия отмены/коррекции.
- **Что реализовано частично / поверхностно / фрагментарно:** Верхний контекст и mobile/desktop ритм ещё не полностью синхронизированы с section-first каноном.
- **Что не реализовано:** Полный product-grade цикл локальных ошибок, correction/undo ergonomics и визуальной плотности (без semantic drift).
- **Какие legacy / compatibility / structural debt мешают:** Остатки form-first логики и совместимость старого маршрута `/operation`.
- **Шаг утверждённого плана для доведения:** 4.4, 4.5, 4.6.
- **Какие подшаги логично ожидаются позже:** Дошлифовать compact states, расширить быстрые паттерны выбора позиции, закрыть mobile parity.
- **Будущие доказательные артефакты глубокой реализации:** Набор e2e на row rhythm (desktop+mobile), UX state matrix для операций.

### 1.8 Explainability-слой

- **Название канона:** Explainability-слой (почему система показала именно это состояние).
- **Канонический смысл:** Пользователь и администратор должны видеть основания состояния: источник факта, статус, причину отмены/коррекции, применённые правила и последствия для проекций.
- **Какие части repo уже относятся к канону:** `AuditLog` в `prisma/schema.prisma`, history pages/components, recovery consistency contracts, admin import jobs.
- **Что уже реализовано содержательно:** Базовый аудит и история фактов существуют; recovery возвращает structured outcomes.
- **Что реализовано частично / поверхностно / фрагментарно:** Объяснимость фрагментирована между экранами; нет единого explainability UX/diagnostics языка.
- **Что не реализовано:** Централизованный explainability dashboard/trace в control plane с единым доступом к причинам и последствиям.
- **Какие legacy / compatibility / structural debt мешают:** Разные форматы сообщений/ошибок и отсутствие общего слоя сигналов.
- **Шаг утверждённого плана для доведения:** 6.1–6.3, 9.1.
- **Какие подшаги логично ожидаются позже:** Ввести унифицированные trace views для движения/импорта/recovery, стандартизировать формулировки причин.
- **Будущие доказательные артефакты глубокой реализации:** Единая страница диагностики и объяснения, контрактные тесты explainability API.

### 1.9 App shell / поиск / навигация

- **Название канона:** App shell, поиск и навигационный каркас.
- **Канонический смысл:** Навигация должна выражать единую операционную систему: целостный shell, согласованный role-aware доступ, быстрый переход между режимами чтения одной истины.
- **Какие части repo уже относятся к канону:** `src/components/layout/*`, `src/lib/navigation.ts`, route-группа `src/app/(app)/*`, mobile nav, ui-texts.
- **Что уже реализовано содержательно:** Единый shell, desktop sidebar и mobile bottom nav реализованы; ключевые рабочие разделы присутствуют.
- **Что реализовано частично / поверхностно / фрагментарно:** Полная shell coherence (включая глубину role-aware видимости и единый operational rhythm) не завершена.
- **Что не реализовано:** Финальный premium navigation/story-state, где control plane и операционный контур ощущаются как одна система.
- **Какие legacy / compatibility / structural debt мешают:** Остатки совместимости старых маршрутов и неравномерная зрелость модулей второго слоя.
- **Шаг утверждённого плана для доведения:** 4.5–4.6, 5.1–5.3, 7.2–7.3.
- **Какие подшаги логично ожидаются позже:** Довести role-aware меню до полного соответствия канону; синхронизировать mobile «Ещё» с desktop информационной архитектурой.
- **Будущие доказательные артефакты глубокой реализации:** Сценарные e2e на навигационные карты ролей + UX чек-лист shell coherence.

### 1.10 Роли и доступы

- **Название канона:** Роли и доступы.
- **Канонический смысл:** Ролевой доступ задаёт не только security-guard, но и управляемую видимость возможностей в пользовательском и административном контурах.
- **Какие части repo уже относятся к канону:** `Role` в `prisma/schema.prisma`, `src/lib/auth/*`, API guards, admin routes `src/app/admin/*` и `src/app/api/admin/*`, e2e auth setup.
- **Что уже реализовано содержательно:** Серверные guards и role-модель реализованы; административные поверхности выделены.
- **Что реализовано частично / поверхностно / фрагментарно:** Видимость действий и навигации по ролям не везде доведена до единообразного product-level поведения.
- **Что не реализовано:** Полный control-plane policy слой с диагностикой доступов и операционными recovery-действиями по ролям/политикам.
- **Какие legacy / compatibility / structural debt мешают:** Разрыв между security enforcement и UX-видимостью возможностей.
- **Шаг утверждённого плана для доведения:** 5.1–5.3, 6.1–6.3.
- **Какие подшаги логично ожидаются позже:** Явная матрица «роль → модуль → действие → видимость»; сценарии админ-аудита доступов.
- **Будущие доказательные артефакты глубокой реализации:** Контрактная матрица RBAC, e2e тесты видимости/запретов, журналы policy-decisions.

## 2) Надстроечные каноны второго круга

### 2.1 Нативная масштабируемость

- **Название канона:** Нативная масштабируемость.
- **Канонический смысл:** Добавление новых модулей/проекций/интеграций не должно ломать ядро истины и операционный ритм; расширение идёт через канонические контракты, а не through ad-hoc патчи.
- **Какие части repo уже относятся к канону:** Модульные API-роуты, read-model registry/contracts, application use-cases, канонические docs/ADR.
- **Что уже реализовано содержательно:** Foundation для расширения есть (слоистость domain/application/read-model, registry проекций).
- **Что реализовано частично / поверхностно / фрагментарно:** Формальные правила масштабируемого добавления модулей пока не закреплены как завершённый operational contract.
- **Что не реализовано:** Закрытый extension governance (стандарты модулей, критерии качества, обязательные контракты).
- **Какие legacy / compatibility / structural debt мешают:** Наличие transitional compatibility-слоя повышает стоимость масштабирования.
- **Шаг утверждённого плана для доведения:** 8.3, 9.3.
- **Какие подшаги логично ожидаются позже:** Ввести template добавления модуля (domain/write/read/control/tests/docs), обязательный audit-gate для новых расширений.
- **Будущие доказательные артефакты глубокой реализации:** Документ extension contract + эталонный новый модуль, прошедший все канонические gate-checks.

### 2.2 Глубокий аналитико-стоимостной слой

- **Название канона:** Глубокий аналитико-стоимостной слой.
- **Канонический смысл:** Поверх базового операционного учёта формируются расширенные аналитики и cost-представления, но только как проекции канонического факта.
- **Какие части repo уже относятся к канону:** Текущий consumption report (`src/app/(app)/reports/consumption/*`, `src/app/api/reports/consumption/*`), базовые аналитические сущности в схеме, `src/lib/read-models/consumption-report-projection.ts`.
- **Что уже реализовано содержательно:** Базовый отчётный контур есть и связан с каноническими осями.
- **Что реализовано частично / поверхностно / фрагментарно:** Аналитическое покрытие узкое; отсутствует расширенный управленческий набор проекций.
- **Что не реализовано:** Полный deep analytics/cost слой (матрицы, drill-downs, расширенные сигналы, cross-cutting cost views).
- **Какие legacy / compatibility / structural debt мешают:** Историческая фокусировка на одном отчёте и неполная консолидация read-side блока 5.
- **Шаг утверждённого плана для доведения:** 5.3, 8.2.
- **Какие подшаги логично ожидаются позже:** Расширить модель отчётов и фильтров; добавить explainable drill-down до событий и строк.
- **Будущие доказательные артефакты глубокой реализации:** Пакет новых проекций/отчётов + контрактные тесты аналитики и сверки с write-side truth.

### 2.3 Технический / SLA / устойчивостный канон

- **Название канона:** Технический, SLA- и устойчивостный канон реализации.
- **Канонический смысл:** Продукт должен иметь наблюдаемость, диагностируемость, гарантии восстановления и parity между repo/deploy/docs.
- **Какие части repo уже относятся к канону:** health routes (`/api/health*`), recovery service, deploy/runbook docs, import hardening foundation, notification/import job logs.
- **Что уже реализовано содержательно:** Есть зачатки health/recovery/operational документации и механизмов.
- **Что реализовано частично / поверхностно / фрагментарно:** Нет полного observability+SLA контура и системного регрессионного контракта зрелого уровня.
- **Что не реализовано:** Завершённая эксплуатационная зрелость (SLO, алерты, нагрузочные/регрессионные гарантии, подтверждённая parity).
- **Какие legacy / compatibility / structural debt мешают:** Разрозненность сигналов, частичная product-grade нормализация ошибок, зависимость от незавершённого control plane.
- **Шаг утверждённого плана для доведения:** 3.5–3.6, 6.3, 9.1–9.3.
- **Какие подшаги логично ожидаются позже:** Ввести operational scorecard, автоматизировать проверки parity, закрепить SLO/SLI по критическим контурам.
- **Будущие доказательные артефакты глубокой реализации:** Dashboards/alerts, формальный SLA runbook, регрессионный пакет стабильности на CI.

## 3) Глобальные системные разрывы

1. **Legacy naming и compatibility leakage.**
   - Проблема: внутренние `item*`/`purpose*` и часть compatibility-путей ухудшают прозрачность канона в коде и контрактах.
   - Почему системно: затрагивает доменную модель, UI-copy, импорт, тесты и onboarding команды.

2. **Разрыв roadmap-глубины и фактической product-готовности контуров.**
   - Проблема: foundation по многим блокам есть, но продуктовая глубина (UX/error/diagnostics/control) неполная.
   - Почему системно: влияет на корректность принятия решений о «готовности» этапов.

3. **Разрыв docs ↔ code ↔ deploy parity.**
   - Проблема: часть пользовательских и эксплуатационных эффектов зависит от того, насколько актуален деплой относительно repo.
   - Почему системно: снижает доверие к наблюдаемому состоянию и к проверкам на соответствие канону.

4. **Неполный control plane как единый управляющий слой.**
   - Проблема: есть полезные административные модули, но нет цельного root control plane с unified visibility/policies/diagnostics/recovery actions.
   - Почему системно: блокирует explainability и operational maturity сразу для нескольких канонов.

5. **Неполная shell coherence и роль-ориентированная навигационная зрелость.**
   - Проблема: app shell работает, но не полностью закрывает канонический ритм единой операционной системы на всех ролях и устройствах.
   - Почему системно: мешает глубокой реализации хаба «Движения» и базового продуктового контура.

6. **Незавершённая operational зрелость.**
   - Проблема: health/recovery foundation есть, но нет финального SLA/observability/regression контракта.
   - Почему системно: ограничивает управляемость эксплуатации и качество масштабирования.

7. **Второй круг канонов пока в foundation-режиме.**
   - Проблема: масштабируемость, deep analytics/cost и технический канон описаны, но реализованы частично.
   - Почему системно: расширения рискуют идти поверх незакрытого базового продукта.

## 4) Карта зависимости канонов

### 4.1 Фундаментальные зависимости

- **Атом доменного факта** → фундамент для всех остальных канонов.
- **Доменные оси** и **единицы измерения** → обязательная база для корректных движений, отчётов и explainability.
- **Временная модель** + **жизненный цикл движения** → фундамент для recovery, истории, SLA и контрольных действий.

### 4.2 Что нельзя полноценно реализовать до очистки модели

- Без закрытия 1.3/1.4 (термины и compatibility-layer) нельзя считать завершёнными:
  - строку позиции учёта как чистый operational язык;
  - shell coherence;
  - explainability и роль-ориентированную читаемость.

### 4.3 Что зависит от завершения хаба «Движения»

- Полноценная реализация канонов:
  - строка позиции учёта;
  - app shell / поиск / навигация;
  - часть explainability на уровне ежедневного ритма действий.
- Критические зависимости: подэтапы 4.4–4.6.

### 4.4 Что зависит от control plane

- Полноценная реализация:
  - роли и доступы (как управляемый policy-слой, а не только guards);
  - explainability-слой (единая диагностика и trace);
  - технический/SLA/устойчивостный канон (операционный контроль и recovery actions).
- Критические зависимости: блок 6 + 9.

### 4.5 Что относится ко второму кругу и идёт после закрытия базового продукта

- **Нативная масштабируемость** (8.3) — после стабилизации базового контура 1–6.
- **Глубокий аналитико-стоимостной слой** (5.3, 8.2) — после консолидации ядра read-side и терминологии.
- **Технический/SLA/устойчивостный канон** в финальной форме (9.1–9.3) — после сборки control plane и подтверждения parity.
