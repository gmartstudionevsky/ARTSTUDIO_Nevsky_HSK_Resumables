# Аудит соответствия итоговой спецификации ARTSTUDIO Consumables

Дата: 2026-03-06

## Методика проверки

- Анализ схемы данных, API-эндпоинтов и ключевых UI-компонентов.
- Проверка базовых инженерных проверок (`lint`, `typecheck`).
- Попытка запуска e2e smoke-теста (не выполнен из-за отсутствия браузера Playwright в среде).

## Краткий итог

- Ядро MVP реализовано: роли, логин, операции IN/OUT/ADJUST, склад, история, инвентаризация, каталог, справочники, Telegram, отчёт расхода + экспорт CSV/XLSX.
- Есть существенные расхождения с зафиксированной спецификацией в части учётной модели, UX потока исправлений и полноты отчётности/фильтров.

## Что НЕ реализовано или реализовано не по спецификации

### 1) Инвентаризация применяется не как `INVENTORY_APPLY`

По спецификации применение инвентаризации должно формировать batch типа `INVENTORY_APPLY`, но фактически создаются транзакции типов `IN` и `OUT`.

### 2) Открытие склада не проводится отдельным типом `OPENING`

В интерфейсе есть режим «Открытие», но при применении создаётся `IN` с заметкой «Открытие склада 01.03.2026», а не операция типа `OPENING`.

### 3) Создание позиции с «сразу есть количество» не завершено

В форме есть UX-блок «Товар уже на складе», но значение количества/единицы не отправляется в API создания позиции и приход автоматически не создаётся.

### 4) Нет полноценного Undo после сохранения операции

Есть toast «Операция записана», но без действия отмены в короткое окно времени (undo-action).

### 5) В результате сохранения нет кнопки «Создать ещё операцию»

Экран результата даёт отмену batch/строки и исправление строки, но отдельной быстрой кнопки «Ещё» нет.

### 6) Быстрые действия в истории не выведены на список

Спецификация требует быстрые действия «отменить/исправить» в истории как рабочем инструменте. Сейчас в списке только переход «Открыть» в детали.

### 7) Фильтры истории не полные относительно спецификации

Отсутствуют отдельные фильтры по пользователю и разделу (категории); есть поиск, тип, статус, позиция, статья, назначение.

### 8) Отчётность реализована частично

Обязательный отчёт «расход по статьям → позиции» есть, но отсутствуют:

- матрица статья × назначение;
- топ расходников;
- отдельный складской отчёт «ниже минимума/ноль/минус» как отчётный экран;
- отчёт по инвентаризациям (отклонения/кто применял/комментарии);
- drill-down до операций/строк из отчёта расхода.

### 9) Нет сохранённых фильтров (desktop)

Для склада/истории в спецификации предусмотрены сохранённые фильтры. В коде применяются только текущие локальные состояния фильтров.

### 10) Нет блока «недавние/частые позиции» в операции

В операции присутствуют поиск и выбор позиции, но отсутствуют отдельные быстрые группы recent/frequent.

### 11) Нет autocomplete-поля выбора позиции как основной паттерн

Сейчас реализована связка «строка поиска + select», а не полноценный autocomplete с контекстными подсказками в одном контроле.

### 12) Редактор UI-текстов без предпросмотра

Сущность UI-текстов и scope есть, но отсутствует «предпросмотр, как будет выглядеть».

### 13) Telegram-сообщения не полностью соответствуют типам операций

Шаблон Telegram классифицирует тип как «Приход/Расход/Коррекция» и не покрывает явно `OPENING`/`INVENTORY_APPLY`.

### 14) Telegram daily digest не поддерживает минусовые остатки как опцию

В правилах digest есть ниже минимума/нулевые, но не настраиваемый сигнал по отрицательным остаткам.

### 15) Навигация desktop не ограничена по ролям на уровне списка меню

Desktop-меню содержит пункты «Номенклатура» и «Отчёты» для всех пользователей; доступ ограничивается уже на уровне страниц. Это не нарушение безопасности, но UX-расхождение с «жёсткой» моделью видимости возможностей.

## Что реализовано корректно (ключевое)

- Ролевая модель `SUPERVISOR` / `MANAGER` / `ADMIN` и серверные guard-ы по ролям.
- Принцип «истина в операциях»: остатки считаются через агрегирование транзакций/строк.
- Статусы `ACTIVE/CANCELLED` без физического удаления; есть причины/комментарии отмены и аудит.
- Исправление строки через отмену исходной и создание новой с `correctedFromLineId`.
- Мульти-единицы позиции с проверкой allowed + ровно одна default input/report среди разрешённых.
- Деактивация/реактивация позиции и фильтрация активности.
- Импорт XLSX + отдельный раздел админки.
- Отдельный мобильный bottom navigation.

## Зафиксированные инциденты (дополнительно)

- Ошибка при импорте из таблицы:
  - `Invalid prisma.purpose.upsert() invocation: Transaction API error: Transaction not found. Transaction ID is invalid, refers to an old closed transaction Prisma doesn't have information about anymore, or was obtained before disconnecting.`
- Требуется отдельный технический разбор импорта с акцентом на устойчивость длинных транзакций Prisma, корректный жизненный цикл tx-контекста и безопасную обработку ретраев/батчей.
