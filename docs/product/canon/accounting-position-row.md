# Строка позиции учёта

## 1. Статус документа

Этот документ фиксирует отдельный нормативный уровень «Строка позиции учёта» в ARTSTUDIO Consumables.

Документ является седьмым документом очереди канонизации после канона атома доменного факта, канона доменных осей и пространственной модели, канона единиц измерения, канона временной модели, канона жизненного цикла движения и канона специальных режимов действия.

## 2. Назначение канона строки позиции учёта

Канон строки позиции учёта задаёт explainable и practically strong модель основной рабочей единицы хаба «Движения».

Данный канон:
- закрепляет строку как центральную прикладную единицу действия;
- не допускает смешения строки интерфейса, позиции учёта, подготовленного движения и атома доменного факта;
- задаёт основу quantity-first взаимодействия;
- задаёт каноническое место единицы измерения, комментария, сигналов и скрытых расширений;
- делает базовый рабочий поток быстрым, ясным, low-noise и explainable;
- создаёт мост между доменной моделью и реальным поведением главного рабочего экрана.

## 3. Определение строки позиции учёта

Строка позиции учёта — это каноническая прикладная рабочая единица хаба «Движения».

В строке одна позиция учёта представлена как непосредственная точка пользовательского действия, чтения контекста, ввода количества, выбора рабочей единицы измерения, добавления комментария, чтения сигналов и включения изменения в общий проход по разделу.

Строка не является минимальным доменным фактом.
Строка не является самостоятельной доменной сущностью.
Строка является канонической UX-формой работы с позицией учёта внутри основного операционного пространства продукта.

## 4. Каноническая роль строки в системе

Фиксируются следующие положения:
- строка является центральной прикладной единицей normal flow внутри хаба «Движения»;
- пользователь работает со строками как с живыми рабочими ячейками действия;
- через строку пользователь видит позицию, вводит количество, понимает единицу измерения, при необходимости добавляет комментарий и накапливает изменение;
- строка концентрирует enough context для уверенной работы;
- строка является достаточно сильной, чтобы минимизировать menu-diving в обычной работе.

## 5. Принцип row is operational unit, not domain atom

Фиксируются следующие правила:
- строка позиции учёта является прикладной операционной единицей, но не атомом доменного факта;
- строка не равна движению как доменной записи;
- строка не равна атому доменного факта;
- строка не является самостоятельной доменной сущностью вне позиции и рабочего контекста;
- строка может подготовить один или несколько будущих атомарных фактов, но не подменяет их собой.

## 6. Принцип one row, one position context

Фиксируются следующие правила:
- одна строка выражает один ясный рабочий контекст одной позиции учёта;
- строка не является контейнером нескольких независимых позиций;
- позиция однозначно опознаваема в строке;
- одна строка — одна основная предметная единица действия в конкретном разделе и рабочей сцене.

## 7. Принцип quantity-first interaction

Фиксируются следующие правила:
- строка строится вокруг quantity-first interaction;
- главным действием строки является ввод количества;
- всё остальное подчинено этому ядру;
- количество является visually and interactionally central;
- пользователь с первого взгляда понимает, где происходит основное действие.

## 8. Принцип visible unit, hidden complexity

Фиксируются следующие правила:
- рабочая единица измерения нативно видна рядом с количеством;
- смена рабочей единицы является лёгкой и explainable;
- вторичная техническая или контекстная сложность не захватывает базовую поверхность строки;
- строка остаётся low-noise even when functionally rich.

## 9. Принцип progressive detail without leaving the row

Фиксируются следующие правила:
- вторичные действия и расширения по возможности открываются внутри строки или как её лёгкое расширение;
- пользователь не проваливается в отдельный экран ради обычного комментария или уточнения;
- базовая сцена normal flow сохраняется;
- строка является enough-rich для практической работы и не превращается в карточку со всем миром внутри.

## 10. Каноническая структура строки позиции учёта

Строка строится из следующих зон:
- зона идентичности позиции;
- зона основного количества;
- зона рабочей единицы измерения;
- зона вторичных действий и комментария;
- зона сигналов и статуса строки;
- зона скрытого расширения, если оно раскрыто.

На desktop и mobile композиция может различаться.
Смысловое присутствие этих зон сохраняется на обеих платформах.

## 11. Канон зоны идентичности позиции

Фиксируются следующие правила:
- зона нужна для безошибочного понимания, с какой позицией учёта ведётся работа;
- в зоне читается основная идентичность позиции;
- в зоне читается practically relevant наименование;
- при необходимости читается краткий дополнительный контекст;
- идентичность остаётся ясной, быстрой для сканирования и не растворяется внутри поля ввода.

## 12. Канон зоны основного количества

Фиксируются следующие правила:
- зона основного количества является смысловым центром строки;
- пользователь вводит количество и видит текущее рабочее значение именно в этой зоне;
- поле количества является immediately visible;
- поле количества быстро доступно и дружелюбно;
- поле количества не спрятано за вторичными действиями.

## 13. Канон зоны рабочей единицы измерения

Фиксируются следующие правила:
- зона не допускает чтение количества как абстрактного числа;
- пользователь видит текущую рабочую единицу;
- пользователь понимает, в чём вводит значение;
- при необходимости пользователь нативно и быстро меняет рабочую единицу в пределах канона позиции;
- единица visually связана с количеством и не является глубокой настройкой.

## 14. Канон зоны вторичных действий и комментария

Фиксируются следующие правила:
- зона поддерживает practically relevant вторичные действия без перегруза normal flow;
- пользователь при необходимости добавляет комментарий;
- пользователь при необходимости открывает вторичное расширение строки;
- пользователь при необходимости читает или уточняет дополнительный контекст;
- комментарий и вторичные действия доступны, но не перехватывают смысловой центр у количества.

## 15. Канон зоны сигналов и статуса строки

Фиксируются следующие правила:
- зона сигналов и статуса поддерживает explainability без текстового шума;
- продукт может показывать подготовленность изменения;
- продукт может показывать локальную проблему или ограничение;
- продукт может показывать important special-context signal и иные relevant status cues;
- сигналы остаются видимыми, лаконичными, explainable и low-noise.

## 16. Канон скрытого расширения строки

Фиксируются следующие правила:
- скрытое расширение допустимо и вторично;
- строка раскрывает дополнительную глубину по запросу;
- базовый рабочий режим не зависит от обязательного раскрытия расширения;
- скрытое расширение не превращает строку в mini-screen с перегруженной логикой;
- вторичная глубина усиливает работу и не заменяет собой normal flow.

## 17. Канон строки и позиции учёта

Фиксируются следующие правила:
- строка не тождественна позиции учёта как доменной сущности;
- позиция существует в модели независимо от конкретного UI-представления;
- строка является прикладной формой доступа к позиции внутри хаба;
- одна и та же позиция может читаться в других режимах продукта;
- строка остаётся explainable через позицию и не подменяет её собой.

## 18. Канон строки и атома доменного факта

Фиксируются следующие правила:
- строка не является атомом доменного факта;
- заполнение строки само по себе не означает существование факта;
- одна строка может породить один атом движения или несколько атомов в зависимости от нормативного маршрута;
- атом факта появляется не в момент отображения строки, а в момент нормативной фиксации движения;
- строка является explainable подготовительным прикладным слоем.

## 19. Канон строки и подготовленного движения

Фиксируются следующие правила:
- строка может содержать подготовленное движение как прикладной слой будущей фиксации;
- пользователь работает со строкой как с местом подготовки будущего изменения;
- подготовленное движение локально выражается через введённое количество и рабочую единицу;
- до нормативной фиксации строка не маскируется как уже проведённое движение;
- строка ясно отделяет подготовленность от уже существующей истории.

## 20. Канон строки и normal flow

Фиксируются следующие правила:
- строка является основной единицей normal flow в хабе «Движения»;
- базовая повседневная работа сотрудника происходит через проход по строкам;
- normal flow собирается как series of row-level actions;
- строка не строится как special mode shell;
- обычная работа ощущается как естественный ритм строк, а не как серия тяжёлых форм.

## 21. Канон строки и special modes

Фиксируются следующие правила:
- базовая строка ориентирована на normal flow;
- при связи строки с correction, inventory-driven context или иным special-origin signal это показывается explainably;
- special mode не растворяется скрыто в обычной строке;
- special-mode presence не превращает строку в административный overload.

## 22. Канон строки и жизненного цикла движения

Фиксируются следующие правила:
- строка лаконично отражает relevant lifecycle-смысл, но не превращается в полную state-machine;
- пользователь понимает, есть ли подготовка, проблема, уже существующее связанное движение или иной relevant state;
- строка не обязана показывать всю полноту lifecycle-модели;
- deep lifecycle explainability достижима, но не навязана.

## 23. Канон строки и времени

Фиксируются следующие правила:
- строка показывает время в low-noise user-friendly форме;
- в normal flow пользователю достаточно practically meaningful temporal grain;
- строка не перегружена полной timestamp-онтологией;
- временная explainability доступна там, где требуется;
- строка не разрушает временную модель, даже если показывает только дату без времени.

## 24. Канон строки и единиц измерения

Фиксируются следующие правила:
- рабочая единица ясно видима рядом с количеством;
- смена рабочей единицы является лёгкой и нативной;
- пользователь понимает, в какой единице работает;
- строка не создаёт количественную двусмысленность;
- строка не скрывает метрический смысл ради визуальной чистоты.

## 25. Канон строки и комментария

Фиксируются следующие правила:
- комментарий является вторичным explainability-инструментом и не является смысловым центром;
- комментарий easy-to-add;
- комментарий не тормозит обычный проход;
- комментарий не становится обязательной бюрократической ступенью normal flow;
- строка позволяет комментарий там, где он помогает понять или обосновать действие.

## 26. Канон строки и накопления изменений

Фиксируются следующие правила:
- строка пригодна для сценария накопления изменений внутри одной рабочей сессии;
- пользователь проходит по нескольким строкам и подготавливает несколько изменений;
- каждая строка хранит local prepared state до общего применения;
- строка остаётся читаемой после ввода;
- пользователь понимает, какие строки уже включены в текущую рабочую сессию.

## 27. Канон строки и кнопки «Применить»

Фиксируются следующие правила:
- строка не живёт изолированно от общего механизма применения;
- строка готовит изменения и не обязана быть местом финальной фиксации всей рабочей сессии;
- строка совместима с общей логикой накопления и общего применения;
- local prepared state строки читается в связке с общей сессией;
- кнопка «Применить» завершает серию row-level действий и не подменяет смысл каждой строки.

## 28. Канон строки и explainability

Фиксируются следующие правила:
- строка является explainable by design;
- пользователь понимает, с какой позицией работает;
- пользователь понимает, что именно вводит;
- пользователь понимает, в какой единице работает;
- пользователь понимает, находится ли строка в нормальном состоянии, в подготовке, в ограничении или в special-context;
- UI не превращает строку в красивую, но смыслово непрозрачную единицу.

## 29. Канон строки и mobile

Фиксируются следующие правила:
- mobile-версия строки является самостоятельной сильной формой и не является сжатой desktop-строкой;
- строка остаётся touch-ready;
- quantity input является быстрым и безопасным;
- единица измерения легко читаема и переключаема;
- сигналы и вторичные действия доступны без перегруза;
- mobile-row сохраняет normal-flow power в focused mobile form.

## 30. Канонические состояния строки

Строка имеет собственные прикладные состояния, различимые от жизненного цикла движения:
- спокойная строка;
- редактируемая строка;
- подготовленная строка;
- строка с ограничением;
- строка с предупреждением;
- строка со специальным контекстом;
- строка с раскрытым расширением.

Эти состояния не являются lifecycle движения.
Эти состояния являются operational UI-state строки как рабочей единицы.

## 31. Канон спокойной, редактируемой и подготовленной строки

Фиксируются следующие определения:
- спокойная строка — без локальной проблемы и без незавершённой подготовки;
- редактируемая строка — пользователь работает с количеством, единицей или локальным контекстом;
- подготовленная строка — в строке сформирован local prepared state для включения в общую сессию.

## 32. Канон строки с ограничением / предупреждением / специальным контекстом

Фиксируются следующие правила:
- если строка не может использоваться корректно или требует осторожности, это показывается ясно и лаконично;
- пользователь понимает, что ограничено;
- пользователь понимает, почему ограничено;
- пользователь понимает, может ли продолжить;
- пользователь понимает, нужен ли иной сценарий;
- при связи строки со special-context это видно без разрушения normal-flow поверхности.

## 33. Правило строки как practically sufficient unit

Фиксируются следующие правила:
- строка является practically sufficient для основного объёма повседневной работы;
- normal flow не требует постоянного проваливания за пределы строки;
- строка покрывает типовой набор действий сотрудника;
- отдельные экраны и глубокие формы остаются вторичными.

## 34. Правило low-noise richness

Фиксируются следующие правила:
- строка может быть функционально богатой и остаётся low-noise;
- сигналы, единица измерения, комментарий и расширения допустимы;
- перечисленные элементы не создают визуальный и смысловой шум;
- richness допустим только как controlled richness.

## 35. Правило row-level explainability

Фиксируются следующие правила:
- пользователь понимает, что это за позиция;
- пользователь понимает, что может сделать;
- пользователь понимает, в каком состоянии находится строка;
- пользователь понимает, какие ограничения или сигналы существуют;
- пользователь понимает, не требуется ли special mode для нетипового случая.

## 36. Правило non-substitution of row for fact

Фиксируются следующие правила:
- UI не создаёт видимость, что строка и есть already existing movement;
- lifecycle движения не сводится к локальному состоянию строки;
- история не читается как «состояние строки на экране»;
- строка остаётся прикладным интерфейсным слоем и не является нижней онтологией продукта.

## 37. Недопустимые подмены строки позиции учёта

Нарушением канона признаются:
- строка как якобы доменный атом факта;
- строка, содержащая несколько независимых позиций как одну рабочую единицу;
- скрытый quantity input, уступающий смысловой центр вторичным действиям;
- единица измерения, скрытая так, что пользователь не понимает, в чём работает;
- строка как mini-form со всем возможным функционалом сразу;
- строка, требующая постоянного ухода в дополнительные экраны ради normal flow;
- строка, визуально красивая, но смыслово непрозрачная;
- mobile-строка как сжатая desktop-версия без touch-логики.

## 38. Канонический маршрут работы со строкой

Канонический маршрут включает последовательность:
1. Пользователь видит строку в контексте выбранного раздела.
2. Пользователь быстро распознаёт позицию.
3. Пользователь видит центральное поле количества и текущую рабочую единицу.
4. При необходимости пользователь меняет рабочую единицу и добавляет комментарий.
5. Пользователь подготавливает изменение в пределах строки.
6. Пользователь получает ясный локальный статус строки.
7. Пользователь продолжает проход по другим строкам без потери ритма.
8. Общее применение завершает серию row-level действий без разрушения explainability каждой строки.

## 39. Каноническая формула строки позиции учёта

Итоговая формула:

Строка является канонической прикладной единицей normal-flow работы в хабе «Движения».
Через строку одна позиция учёта становится immediately readable, quantity-centered, explainable и ready-for-action рабочей точкой.
Пользователь видит идентичность позиции и работает с количеством и рабочей единицей измерения.
При необходимости пользователь раскрывает комментарий и вторичный контекст.
Пользователь накапливает изменение в общей сессии.
Строка не смешивается с атомом доменного факта и не превращается в перегруженную административную форму.

## 40. Роль документа в дальнейшем раскрытии

На основе данного документа раскрываются:
- explainability-слой;
- app shell и поисково-навигационный каркас;
- роли и доступы;
- прикладной correction-сценарий;
- component system и interaction model при инженерной проработке.
